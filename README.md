# A GPU-Batched, Discretized Second-Generation Bandelet Transform

Code accompanying the paper *"Parallelizing Second-Generation Bandelet
Transforms: A GPU-Batched Discretization"* (draft, targeting Elsevier
Digital Signal Processing). See [`paper/paper.pdf`](paper/paper.pdf).

This implements a discretized, exactly-invertible realization of
second-generation bandelets (Peyré & Mallat, 2005) built specifically for
batched GPU execution: a reversible 5/3 lifting wavelet transform, a
per-region directional lift restricted to a discrete set of integer pixel
shears (exact, no interpolation), an explicit "skip" option per region, and
a level-synchronous batched formulation of the usual sequential best-basis
quadtree search. Reconstruction is exact (float32 machine precision) on
every device tested: CPU, four GPU generations (Tesla T4, L4, A100,
Blackwell-generation RTX PRO 6000), and TPU.

## Repository layout

```
notebooks/
  gpu_bandlet_transform.ipynb   Main implementation, self-tests, and the full
                                 development story (5 rounds of GPU decode
                                 optimization with real profiler output at
                                 each round, the multi-scale testing that
                                 found and diagnosed a rate-distortion
                                 parameter question, etc.) -- the most
                                 complete single artifact in this repo.
  bandlet_benchmark.ipynb       Colab wrapper for src/bandlet_benchmark.py
                                 (writes the script to disk, runs it, offers
                                 the .npz result for download). Use this to
                                 reproduce the paper's timing and sparsity
                                 numbers on your own GPU(s).
  bandlet_tpu.ipynb             The TPU experiment (Section 9 of the paper):
                                 a static-shape decode redesign for XLA
                                 compilation, verified correct, benchmarked
                                 against the GPU-oriented decode on real TPU
                                 hardware -- a negative result, reported in
                                 full.

src/
  bandlet_benchmark.py          Standalone script containing the complete
                                 transform (byte-identical core logic to the
                                 main notebook) plus the two experiments used
                                 for the paper: size-vs-time timing (CPU and
                                 GPU) and sparsity/coefficient-decay vs. six
                                 classical wavelet families (via PyWavelets).
                                 Saves everything to one .npz per run.

scripts/
  make_result_figures.py        Reproduces the paper's two data-driven
                                 figures (timing, sparsity) and their tables
                                 from one or more .npz files produced by
                                 bandlet_benchmark.py.
  make_diagram_figures.py       Reproduces the paper's two illustrative
                                 figures (integer-shear mechanism, quadtree
                                 partition example) -- these are generated
                                 from real computation, not hand-drawn, and
                                 the script prints the same numbers
                                 (round-trip error, leaf count) quoted in
                                 the paper's captions.

results/
  Place downloaded .npz files here (see "Reproducing the results" below).
  Empty in this repo; results are hardware-specific and regenerated by you.

paper/
  paper.tex, paper.pdf, figs/   The paper itself and all four figures.

tests/
  run_offline_selftest.py       Correctness self-test requiring no GPU (see
                                 below).
  torch_shim/                   A small NumPy-backed stand-in for the subset
                                 of the torch API this project uses. This is
                                 not a general torch replacement -- it exists
                                 because the environment this project was
                                 developed in had no GPU access at all, so
                                 every change was verified against this shim
                                 for logical correctness before being handed
                                 off for a real GPU run. Included for
                                 transparency and so the self-test can run
                                 anywhere.
```

## Quick start

### I just want to read/rebuild the paper
```
cd paper
pdflatex paper.tex && pdflatex paper.tex   # twice, for cross-references
```
Note: this was written and compiled against the standard LaTeX `article`
class, not Elsevier's `elsarticle.cls`, which wasn't available in the
offline environment used to write it. See the comment at the top of
`paper.tex` for the (small) change needed to switch classes for final
submission.

### I want to verify the transform is correct, with no GPU
```
pip install numpy scikit-image
python tests/run_offline_selftest.py
```
Runs 7 checks (lifting DWT, shear, per-leaf transform, quadtree
segmentation across rate-distortion extremes, full pipeline on a real
image) using the included torch shim. All should report `[PASS]` with
errors at floating-point machine precision.

### I want to reproduce the paper's timing and sparsity results on my own GPU
Open `notebooks/bandlet_benchmark.ipynb` in Colab (or run
`src/bandlet_benchmark.py` directly on any CUDA machine):
```
pip install torch PyWavelets scikit-image numpy
python src/bandlet_benchmark.py --out results/results_<your_gpu>.npz \
    --sizes 64,128,256,512,1024,2048 --sparsity-size 512
```
Repeat on as many GPUs as you like, naming each output file distinctly.
Then:
```
python scripts/make_result_figures.py \
    --results results/results_*.npz \
    --out paper/figs
python scripts/make_diagram_figures.py --out paper/figs
```
reproduces every figure and every number tabulated in the paper.

### I want to see the full development story, not just the final code
Open `notebooks/gpu_bandlet_transform.ipynb`. It's substantially longer than
`src/bandlet_benchmark.py` because it keeps the pedagogical and diagnostic
material: the "skip" option ablation, the five-round GPU decode
optimization with the actual `torch.profiler` output that drove each round
(including two rounds that targeted a bottleneck other than the one first
hypothesized), and the multi-scale testing that investigated a leaf-density
question, proposed a fix, found the fix mathematically inert, and traced
the real cause to test-content properties rather than the algorithm — kept
in rather than edited out, because the correction is more useful than a
clean narrative would be.

## Reproducing the TPU result

`notebooks/bandlet_tpu.ipynb` requires a TPU runtime (Colab: *Runtime →
Change runtime type → TPU*) and `torch_xla`. It runs both the GPU-oriented
("dynamic") decode and a static-shape ("static") decode built specifically
for XLA compilation side by side, verifies they produce bit-identical
output, and times both. On the TPU hardware used for the paper, the static
version — despite being the one designed for TPU's constraints — was two to
three orders of magnitude *slower* than the unmodified GPU decode. See
Section 9 of the paper for the analysis.

## Correctness

Every stage of the forward transform (lifting, integer shear, quadtree
merge/split) is either an exact permutation or an exactly invertible
lifting step; there is no lossy approximation anywhere in the pipeline.
This was confirmed empirically on CPU, four GPU architectures, and TPU, at
image sizes from 64×64 to 2048×2048, with maximum reconstruction error
consistent with float32 arithmetic (no error accumulation beyond what's
expected from floating-point rounding) in every case reported in the paper.
